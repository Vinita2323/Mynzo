HIGH
No rate limiting on any endpoint — OTP brute-force possible
app.js — zero rate-limit middleware
▶

There is no express-rate-limit or any throttling anywhere. An attacker can: (1) enumerate any phone number with unlimited OTP guesses, (2) flood your SMS provider with costs, (3) DDoS the entire API, (4) abuse the coupon validation endpoint to enumerate valid codes.

Install & apply immediately
npm install express-rate-limit // app.js const rateLimit = require('express-rate-limit'); // Global: 100 req/15min per IP app.use(rateLimit({ windowMs: 15*60*1000, max: 100 })); // OTP: 5 attempts per 10 minutes const otpLimiter = rateLimit({ windowMs: 10*60*1000, max: 5, message: { success: false, message: 'Too many OTP requests' } }); app.use('/auth/send-otp', otpLimiter); app.use('/auth/verify-otp', otpLimiter); // Admin login: 10 attempts per 15 min const adminLimiter = rateLimit({ windowMs: 15*60*1000, max: 10 }); app.use('/admin/auth/login', adminLimiter);

▶

The socket handler accepts userId directly from the client payload (socket.on('toggle_like', {'{'} userId, productId {'}'} )). Any connected client can pass any userId and manipulate another user's wishlist or trigger events on their behalf. There is no JWT validation on socket connections.

HIGH
Stock race condition — overselling possible under concurrent orders
orderController.js — non-atomic stock decrement
▶

Current flow: (1) Create order, (2) loop through items and update stock. Two concurrent requests for the last item in stock will both succeed, then both decrement — allowing stock to go to -1. This is a textbook race condition and a real-money bug.

Fix — atomic conditional update
// In createOrder, BEFORE creating the order: for (const item of items) { const result = await Product.findOneAndUpdate( { _id: item.productId, stock: { $gte: item.quantity } // Only if enough stock exists }, { $inc: { stock: -item.quantity, sales: item.quantity } }, { new: true } ); if (!result) { return res.status(400).json({ success: false, message: `"${item.name}" is out of stock` }); } } // Then create the order (stock is already reserved)
HIGH
Coupon double-spend race condition
orderController + couponController — validate and increment are not atomic
▶

Coupon validation (validateCoupon) and increment (usage++ in createOrder) are two separate operations. Two concurrent orders with the same coupon will both pass validation, then both increment — allowing a single-use coupon to be used twice.

Fix — atomic findOneAndUpdate
// In createOrder, validate AND increment in one atomic operation: if (couponCode) { const coupon = await Coupon.findOneAndUpdate( { code: couponCode.toUpperCase().trim(), status: 'Active', expiry: { $gt: new Date() }, $expr: { $lt: ['$usage', '$usageLimit'] } }, { $inc: { usage: 1 } }, { new: true } ); if (!coupon) { return res.status(400).json({ success: false, message: 'Invalid or expired coupon' }); } } // Then remove the separate usage increment below
HIGH
No global error handler — unhandled rejections crash the process
app.js — missing error middleware
▶

app.js has a 404 handler but no 500 error middleware. An unhandled promise rejection in any async route will crash the Node.js process in older versions or silently swallow errors in newer ones. There is also no process.on('unhandledRejection') handler.

Add to app.js after all routes
// Global error handler (4 arguments = Express error middleware) app.use((err, req, res, next) => { console.error({ message: err.message, stack: process.env.NODE_ENV === 'production' ? undefined : err.stack, path: req.path, method: req.method, timestamp: new Date().toISOString() }); const status = err.status || err.statusCode || 500; res.status(status).json({ success: false, message: process.env.NODE_ENV === 'production' ? 'Something went wrong' : err.message }); }); // index.js — process-level safety net process.on('unhandledRejection', (reason) => { console.error('Unhandled Rejection:', reason); // Optionally: graceful shutdown }); process.on('uncaughtException', (err) => { console.error('Uncaught Exception:', err); process.exit(1); // Always exit on uncaught exception });

Order.find({}).populate(...) with no limit will load every order ever created. At 10,000 orders this causes memory spikes and timeouts. Same issue in getUserOrders and getUsers.

Load balancers and monitoring tools need a proper health check that verifies both the server AND the database are healthy. The current / endpoint doesn't check MongoDB connectivity.

Add proper health check
// app.js app.get('/health', async (req, res) => { try { await mongoose.connection.db.admin().ping(); res.json({ status: 'ok', db: 'connected', uptime: process.uptime(), timestamp: new Date().toISOString() }); } catch (err) { res.status(503).json({ status: 'error', db: 'disconnected' }); } });
MEDIUM
No graceful shutdown — in-flight requests killed on deploy
index.js — no SIGTERM handler
▶

When the server restarts (deploy, crash), active connections are killed immediately. Orders being placed mid-request can be left in an inconsistent state.

Graceful shutdown
// index.js const shutdown = async (signal) => { console.log(`Received ${signal}, shutting down gracefully...`); server.close(async () => { await mongoose.connection.close(); console.log('Server and DB connections closed'); process.exit(0); }); // Force exit after 10s setTimeout(() => process.exit(1), 10000); }; process.on('SIGTERM', () => shutdown('SIGTERM')); process.on('SIGINT', () => shutdown('SIGINT'));
MEDIUM
No CDN for static assets — all traffic hits the Node.js server
Backend serves /uploads — Node.js is not optimized for static file serving
▶

Node.js/Express is not an efficient static file server. Images served from /uploads consume Node.js event loop threads. All product images, banners, and videos should be served from a CDN (Cloudflare, AWS CloudFront).

CDN + Cache-Control headers
// Short term: add aggressive cache headers app.use('/uploads', express.static('uploads', { maxAge: '7d', // 7-day browser cache etag: true, lastModified: true, setHeaders: (res) => { res.set('Cache-Control', 'public, max-age=604800, immutable'); } })); // Long term: Cloudflare R2 + CDN // Store to R2, serve via cdn.mynzo.com // Node.js handles only API, never static files
LOW
No PM2 / process manager configured — single Node process
package.json — start script is just "node index.js"
▶

Running a bare node index.js in production means: one crash = down. No clustering to use all CPU cores. No automatic restart. No log file management.

Add PM2
npm install -g pm2 # ecosystem.config.js module.exports = { apps: [{ name: 'mynzo-api', script: 'index.js', instances: 'max', // Use all CPU cores exec_mode: 'cluster', watch: false, max_memory_restart: '500M', env_production: { NODE_ENV: 'production' } }] }; pm2 start ecosystem.config.js --env production pm2 save pm2 startup # Auto-start on reboot

Current architecture

React Frontend (PWA) → Express API → MongoDB Atlas
Admin React SPA → Express API → Local disk /uploads
Frontend (Socket.io) → Socket.io (same server) → No auth on socket

Recommended production architecture (handles 10-100x traffic)

Frontend (Vercel/CDN) → Cloudflare CDN → NGINX / Load Balancer → Express API (PM2 cluster)
Express API → Redis (cache + sessions) + MongoDB Atlas (with indexes)
Express API → AWS S3 / Cloudflare R2 (images/videos via CDN URL)
Socket.io → Redis Adapter (required for multi-instance socket sync)
